# AI エージェントがシナリオを開発するときの行動指針

このドキュメントは、AI エージェント（Claude、Copilot、Cursor、その他）が Voreux のシナリオを作成・修正するときに従うべき行動指針です。

対象読者:
- シナリオの実装を AI エージェントに依頼する人
- AI エージェント自身（プロンプトやリポジトリコンテキストとして読む場合）

---

## 1. なぜこの指針が必要か

Voreux のテストは **人間が見て分かる振る舞い** を検証します。
録画・スクリーンショット・VRT はすべて「人間の目に見える変化」を前提にしています。

AI エージェントは DOM テキストやピクセル差分を機械的に読めますが、それが **人間にとって見える変化かどうか** は推論でしかありません。
この区別を曖昧にすると、機械だけが検出できる微小な差分をテストしてしまい、Voreux の目的から外れます。

---

## 2. 作業開始前に確認すること

シナリオの実装に入る前に、エージェントは以下を明示的に確認・報告すべきです。

### 2.1 自分の観測経路を正直に報告する

エージェントが UI を理解する経路は主に 3 つある。作業開始前に、自分がどの経路を持っているかを明示する。

| 観測経路 | 内容 | 例 |
|---|---|---|
| **DOM-based** | HTML 構造、属性、テキストを読んで状態を推論する | `aria-selected="true"` を読んで「選択されているはず」と判断 |
| **Screenshot-based** | 静止画像（スクリーンショット、VRT diff）を視覚的に確認する | diff 画像を見て「ここが赤くなっている」と判断 |
| **Browser-grounded** | ライブブラウザを操作し、画面の変化をリアルタイムに観測する | ブラウザで実際にクリックし、遷移を直接見る |

報告すべき項目:

- 自分はどの観測経路を持っているか
- ライブブラウザを操作できるか、それともコードを書くだけか
- スクリーンショットや diff 画像を直接見て判断できるか
- テストを自分で実行して結果を確認できるか

**能力が不足している場合はそう言う。** 「見えている」と言いながら実際には DOM テキストから推論しているだけなら、それを明記する。

### 2.2 対象コンポーネントの human-visible 特性を確認する

実装の前に:

- 対象 UI の状態変化は、録画やスクリーンショットで **人間が自然に気づく** レベルか
- それとも、背景色の微妙な変化やサブピクセルの差のように、**拡大しないと分からない** レベルか

この判断を明示的に書き出してから実装に進む。

---

## 3. 段階的な作業ループ

シナリオを書くときは、一度に全部を実装せず、以下のループを繰り返す。

```
1. 現在の状態を調べる（DOM 構造、既存スクリーンショット、ページの見た目）
2. 一つのアクションを実行する（クリック、入力、遷移など）
3. 焦点を絞った証拠を取得する（スクリーンショット、VRT diff、DOM 変化）
4. 何が視覚的に変わったかを説明する
5. その変化が人間に見えるレベルかを判断する
6. 判断に基づいて次に進むか、方針を修正する
```

### 証拠の取得はエージェントの責任

ステップ 3「焦点を絞った証拠を取得する」はユーザーに委ねるものではない。
エージェントが自分の環境で証拠を取得する手段を持っている場合（テスト実行、スクリーンショット確認、既存画像ファイルの閲覧など）は、**自分でそれを実行してから判断する**。

「テストを実行できないのでユーザーに依頼します」は、本当にエージェント側に実行手段がない場合にのみ許される。実行手段があるのにユーザーに返すのは、判断の回避であり、この指針に反する。

### やってはいけないこと

- 全ステップを一気に書いて最後にまとめて確認する
- DOM の属性変化だけを根拠に「見た目が変わった」と主張する
- VRT を実行せずに「差分が出るはず」と仮定する
- 証拠を取得する手段があるのに、取りに行かずユーザーに返す

### やるべきこと

- 各ステップの後に「人間の目にどう見えるか」を一文で説明する
- 説明が書けない、または「微妙な変化」としか言えない場合は、そのステップの妥当性を再検討する

---

## 4. 「見えている」と「推論している」を区別する

エージェントが UI の状態を判断するとき、情報源には信頼度の階層がある。

| 観測経路 | 信頼度 | 例 |
|---|---|---|
| Browser-grounded（ライブブラウザで直接観測） | 最も高い | 「ブラウザでクリックしたら背景色が変わるのを直接見た」 |
| Screenshot-based（静止画像を直接見た） | 高い（ただしエージェントの画像理解能力に依存） | 「スクリーンショットでボタンが青く変わっているのが見える」 |
| DOM / aria 属性をテキストとして読んだ | 中程度（見た目との対応は推論） | 「`aria-selected="true"` が付いたので選択状態になったはず」 |
| VRT の mismatchRatio を数値として読んだ | 低い（数値だけでは人間に見えるかは分からない） | 「mismatchRatio が 0.001 なので変化があった」 |

### 観測経路の優先順位

human-visible かどうかの判断では、より信頼度の高い経路を優先する。

1. **Browser-grounded が使えるなら最優先で検討する。** ライブブラウザでの操作と観測は、人間が実際に体験する状態に最も近い。特に tricky な UI（subtle な色変化、アニメーション、ホバー状態など）では、静止画やDOMテキストでは判断しにくい変化を直接確認できる。
2. **Browser-grounded がない場合は、screenshot-based で補う。** テストを実行して生成されたスクリーンショットや diff 画像を直接確認する。
3. **DOM-based や VRT 数値だけでは、human-visible の判断根拠にはならない。** 補助情報としては有用だが、これらだけで「見える」と結論づけてはいけない。

これは framework の要件ではなく、エージェントの observation capability に関する指針である。Browser-grounded な観測経路を持っていないエージェントもいる。持っていない場合は screenshot-based + staged loop で代替し、そのことを明記する。

### ルール

1. DOM やピクセル数値だけを根拠にする場合は、「推論である」と明記する。スクリーンショットを直接見て判断した場合のみ「確認した」と言ってよい。Browser-grounded で確認した場合は「ライブブラウザで確認した」と言ってよい。
2. **推論のラベリングは免罪符ではない。** 「これは推論です」と書いても、推論だけで完了判断（revise / drop / ship）を出してはいけない。観測で裏付けられない判断は保留し、証拠を取得してから再判断する。
3. 観測可能な証拠が存在するのに取りに行かず、推論で判断を完了させるのは、この指針に反する。
4. より信頼度の高い観測経路が使えるのに、低い経路だけで判断を済ませるのは避ける。

## 5. VRT の閾値と human-visible の関係

### 微小な差分を「人間に見える」と扱わない

pixelmatch の `mismatchRatio` が非常に小さい値（例: `0.00005` = 0.005%）の場合、その差分は:

- サブピクセルレンダリングの揺れかもしれない
- フォントアンチエイリアシングの違いかもしれない
- 録画の圧縮で消えるかもしれない

このレベルの差分を「見た目が変わった証拠」として使うのは不適切です。

### 目安

以下は大まかな感覚値であり、固定的なルールではありません。
実際の判断は **コンポーネントのサイズ、clip 領域の大きさ、変化の種類（色・位置・サイズ）、録画解像度と圧縮** によって大きく変わります。

| mismatchRatio | 一般的な傾向 |
|---|---|
| 非常に小さい（例: < 0.001） | 人間にはほぼ見えないことが多い。テストの根拠としては弱い傾向 |
| 中程度（例: 0.001 〜 0.01） | 注意して見れば分かるかもしれない。変化の種類とコンテキストで判断が必要 |
| 大きい（例: > 0.01） | 人間に見える可能性が高い。テストの根拠として妥当な傾向 |

重要なのは数値そのものではなく、**その差分が「録画を普通の速度で見ている人間に伝わるか」** という問いです。
小さな clip 領域で色が大きく変わる場合は mismatchRatio が小さくても見えることがありますし、大きな領域で位置が微妙にずれる場合は mismatchRatio が大きくても気づきにくいことがあります。

## 6. サンプルの適性を判断する

すべての UI コンポーネントが Voreux のサンプルに適しているわけではありません。

### 良いサンプル候補

- クリックやホバーで **明確に見た目が変わる** コンポーネント
- 状態遷移が **録画で追える** コンポーネント
- 人間が操作の前後を見て「何が起きたか」すぐ分かるコンポーネント

### 避けるべき候補

- 状態変化が微小で、拡大やピクセル比較でしか検出できないコンポーネント
- 変化が内部状態だけで、見た目にほぼ反映されないコンポーネント
- アニメーションのタイミングに強く依存し、スクリーンショットでは捉えにくいコンポーネント

### エージェントに求める行動

**サンプルが human-visible な interaction progression を生み出せないと判断した場合は、無理にテストを書かず、その旨を報告する。** 弱いテストを提出するより、サンプルの再検討を提案する方が価値がある。

---

## 7. フレームワーク API をまず検討する

Voreux は VRT やスクリーンショットのための API を提供しています。

- `ctx.assertNoVisualRegression(baselineName)`
- `ctx.saveCurrentBaseline(baselineName)`
- `ctx.screenshot(name)`
- `ctx.annotateClick(x, y, label)`

新しいシナリオを書くときは、まずこれらの API で要件を満たせないか検討する。

ただし、このリポジトリではサンプルごとにローカルヘルパーを育てる文化があります（`AGENT.md` の「Grow helpers in this order」参照）。フレームワーク API では表現しきれないケース（特殊な clip 領域、コンポーネント固有の比較ロジックなど）では、カスタム実装も適切な選択肢です。その場合は:

- なぜフレームワーク API では不十分かをコメントで説明する
- カスタム実装がサンプルグループ内で再利用できる形にする
- 将来フレームワークに昇格する可能性を意識しつつ、急いで抽象化しない

---

## 8. 判断の完了条件

シナリオの revise / drop / ship を判断するとき、以下の条件を満たしていなければ判断を保留する。

### 判断を出してよい条件

- human-visible かどうかの判断が、**スクリーンショットまたは diff 画像の直接確認**に基づいている
- 取得可能な証拠（既存のスクリーンショット、テスト実行結果、diff 画像など）を**実際に取得した後**である
- 推論のみに基づく判断ではない

### 判断を保留すべき条件

- 証拠を取得する手段があるのにまだ取得していない
- DOM テキストや VRT 数値だけが根拠で、視覚的な確認をしていない
- 「おそらく見えるだろう」「蓋然性が高い」のような推論が判断の中核を占めている

保留した場合は、何の証拠が不足しているかを明示し、次のステップとして証拠の取得を行う。ユーザーに返すのは、エージェント側に取得手段が本当にない場合のみ。

---

## 9. チェックリスト

シナリオの実装が終わったら、以下を確認する。

- [ ] 各ステップの状態変化が、人間の目に見えるレベルだと判断した根拠を説明できるか
- [ ] その根拠はスクリーンショットの直接確認に基づいているか（推論のみなら、証拠を取りに行ったか）
- [ ] 取得可能な証拠（既存画像、テスト実行結果）をすべて確認したか
- [ ] VRT の閾値が human-visible な変化を反映しているか（微小すぎないか）
- [ ] フレームワークの既存 API を使えるところは使っているか
- [ ] `selfHeal` の設定が適切か（初期遷移は `selfHeal: false`）
- [ ] 録画で見たときに、操作の流れが人間に追えるか（annotateClick 等を適切に使っているか）

---

## 10. まとめ

Voreux のシナリオを書くエージェントに求められるのは、コードを正しく動かすことだけではありません。

**「このテストは、人間が見て意味のある変化を検証しているか」** を常に問い続けることです。

機械だけが検出できる差分をテストしても、Voreux の目的には合いません。
自分の能力の限界を正直に報告し、判断の根拠を明示し、サンプルが目的に合わないなら立ち止まる。
それが、このリポジトリで良いシナリオを生み出すための行動指針です。
